Matching: a small PWA for the messy part of getting a game together

How a mobile-first web app turns “who is in?” into a confirmed line-up — with groups, waitlists, payments, team splits, efficient Supabase read paths, and web push.

Part 1The problem is coordination, not scheduling

A recurring football, padel, or board-game group rarely struggles to put a date in a calendar. It struggles with the social state around that date: who is actually in, whether there are enough people, who dropped out, who can bring a +1, and whether everyone has paid.

A group chat works for discussion, but not for keeping an accurate list. “I’m in” is buried under twenty messages. A late cancellation changes the line-up but not the old replies. The organiser ends up tracking everything, answering the same questions, and manually promoting someone from the waitlist.

Matching keeps that information in one place. Organisers create groups and matches. Players join from a link, see the same line-up, land on a waitlist when the match is full, bring an invited +1 when allowed, and receive the changes that matter without reading the whole chat history.

Matching stays focused on the work between sending an invitation and having a dependable line-up. It is not trying to replace the group chat around the game.

Part 2A match is a small state machine

A match is more than an event with a list of names. It has a minimum and maximum player count, a planned/confirmed/cancelled status, optional waitlisting, a notice period for leaving, an optional payment link, and an access rule: group-only or anyone with the link.

Membership has state too. A player can be enrolled, waitlisted, or abandoned. The database trigger that handles capacity makes the transition atomic: the same join cannot be accepted by one screen and rejected by another. When a place opens, the shared state — not a stale chat message — remains authoritative.

Once those states are in place, the organiser can confirm when the minimum is reached, share a line-up, split enrolled players into teams, attach a payment link, and transfer ownership. Players can add the match to their device calendar. Dates are stored with a match timezone, and interface and notification copy are localised in English, French, and Portuguese.

Part 4The stack

Matching is a Next.js App Router application written in TypeScript. React Server Components load the initial page, server actions handle mutations, and small client components handle browser features such as push subscription, touch interactions, dialogs, and navigation feedback.

There is no separate API server. Next.js handles the web routes and mutations, while Postgres handles joined reads and the rules that must still hold when several people join at once.

Part 5Keeping Supabase traffic low

Supabase counts data leaving the database as egress. Broad nested selects and N+1 queries make pages slower and repeatedly send fields that the page never uses.

Pages that load often use Postgres functions that return only what the page needs. get_upcoming_matches returns dates, status, group, venue, counts, and the current player’s status. get_match_details assembles the organiser, venue, group, and ordered enrolled/waitlisted/abandoned lists. Group details, profile metrics, transfer candidates, invitation previews, and public share metadata use the same approach.

Caching has a smaller job. React’s cache() memoises getUser() within a server render, so layout and page code do not validate the same session repeatedly. The public match teaser is also shared between metadata generation and page rendering. Match data itself is not kept in a long-lived cache, because the line-up should update immediately when somebody joins or leaves.

Between June 25 and July 18, Supabase reported 33 MB of total egress and no cached egress. Matching does not use Supabase Storage, Realtime, or Edge Functions: static assets ship with the web app, and notification delivery runs through the Next.js route.

Since Postgres statistics were reset on April 16, 2025, the three busiest screen RPCs have served 58,551 calls. Their average database execution times are approximately 13.2 ms for match detail, 8.2 ms for public metadata, and 6.9 ms for upcoming matches.

I use cache() to avoid repeated work during one render, RPCs to reduce round trips and response size, and a fresh read after a mutation. Each one solves a different problem.

RPCs are not a permission shortcut. Authenticated functions use auth.uid() or has_match_access(...), tables retain row-level security, and executable functions are granted only to the roles that need them. The public metadata function returns only the small amount of match information needed for a shared link.

Part 6Making the PWA work well on phones

Matching is mainly used on phones, often while somebody is on the way to a venue. In the latest week, 92% of visitors were on mobile and seven in ten were on iOS. The PWA manifest launches /dashboard in standalone mode, with app icons, theme-aware chrome, and iOS launch images. The interface uses one controlled scroll region inside 100dvh, which avoids competing page and panel scroll areas.

Part 7Web push notifications

Keeping the line-up accurate is only half the job; people also need to hear when something changes. Matching uses database triggers to call a Next.js notification route when a group match is created, a match is confirmed or cancelled, the time changes, the minimum or maximum is reached, a payment link appears, a player leaves, or ownership transfers.

Recipients opt into group-match and confirmed-match categories separately. The settings screen detects service-worker and PushManager support, exposes a test notification, and can revoke the browser and database subscription together. Translation functions are preloaded once per language for a notification batch, and deliveries run concurrently.

Matching does not ask for notification permission on the landing page. After someone joins a match, a one-time prompt explains that notifications are used for confirmations and changes. Asking at that point gives people some context for the browser permission.

Part 8Lessons from building it

KeepWatch
Keep capacity, waitlisting, and permission checks in Postgres so concurrent joins follow the same rules.Database functions need the same versioning and security review as web routes.
Use RPCs for pages that otherwise need several queries or return too many fields.Simple CRUD is still clearer through the Supabase client.
Memoise repeated server work within a render.Do not cache mutable match data unless invalidation is immediate and reliable.
Account for safe areas, one-handed actions, and standalone launch from the beginning.Installation and push support vary between browsers and operating systems.
Let users choose which notifications they receive.Opening a notification should always load the latest match data.

The stack is familiar on purpose. Most of the work went into keeping the line-up accurate, making the common actions quick on a phone, and notifying people only when something important changes.
