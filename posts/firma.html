Firma: SwiftUI forms where validation and focus share one identity

A source-level tour of mtzaquia/firma — a type-safe form layer that keeps model rules, nested paths, rendered controls, errors, and keyboard focus attached to the same field.

Part 1Why forms drift apart

A form looks like a collection of controls, but its difficult behavior lives in the relationships between them. The text field edits one value, validation writes an error somewhere else, a summary needs to find that error, and submission must scroll to a control that may not even be mounted yet.

Those relationships become fragile when every layer invents its own identity. A model uses writable key paths. Validation uses dictionary keys. a ForEach uses element IDs. SwiftUI focus uses another hashable value. The accessibility layer uses strings. Everything works until a nested optional disappears, a list reorders, or the first invalid row sits outside a lazy container.

Model identityWhich writable property owns the value and its validation rule?
Rendered identityWhich mounted control represents that property right now?
Navigation identityWhere should focus move when rows are nested, reordered, or off-screen?

Firma's design starts by making those the same question. The macro turns a model property into typed field metadata. A FirmaPath extends that metadata through scopes and identified collections. The builder then uses that path for bindings, errors, preferences, scrolling, and FocusState.

The central idea. Firma is less a library of styled controls than an identity layer for editable SwiftUI trees. The convenience controls are useful, but the shared path is what makes validation and focus compose.

Part 2Put editable state and rules in one model

A Firma subject is an observable reference type. Writable properties hold the in-progress values; validate() describes the rules; addError(_:for:) points at a generated field using key-path syntax.

import Firma
import Observation

enum ProfileError: LocalizedError {
  case missingName, invalidEmail

  var errorDescription: String? {
    switch self {
    case .missingName: "Enter your name"
    case .invalidEmail: "Enter a valid email address"
    }
  }
}

@Observable @FormObject
final class ProfileForm {
  var name = ""
  var email = ""
  var receivesUpdates = false

  func validate() {
    if name.isEmpty {
      addError(ProfileError.missingName, for: \.name)
    }
    if !email.contains("@") {
      addError(ProfileError.invalidEmail, for: \.email)
    }
  }
}

This keeps rules close to the state they inspect without putting presentation into the model. The model does not decide whether an error appears below a text field, in a summary, or only after submission. It produces field-addressed errors; the builder and view decide how to present them.

Firma's runtime API is main-actor isolated. That matches SwiftUI's editing and focus machinery, but it is an intentional boundary: a consuming target without main-actor default isolation should mark its form models @MainActor.

Part 3What @FormObject generates

The attached macro requires a class that also has @Observable. For each writable instance property it emits a field descriptor containing the property's label plus getter and setter closures. It also creates one validator per model instance and adds the Firma conformance.

Authored model
@Observable @FormObject final class ProfileForm { var name = "" var email = "" }
→
Conceptual expansion
struct Fields { let name: FirmaField<ProfileForm, String> let email: FirmaField<ProfileForm, String> } static var __fields: Fields let __validator = Validator()

The public-looking expression \.name is therefore not a writable key path into ProfileForm. Context makes it a key path into the generated Fields value, whose leaf is FirmaField<ProfileForm, String>. That indirection gives Firma both compile-time value types and runtime get/set behavior.

The generator deliberately skips constants, static properties, getter-only computed properties, and unsupported binding patterns. Public fields need explicit type annotations because their generated metadata must also be public. Those diagnostics keep an incomplete expansion from turning into obscure errors elsewhere in the builder.

Part 4One path through nested and dynamic forms

A root field can be identified by its label, but composition needs more structure. FirmaPath stores a sequence of field components and actual AnyHashable element IDs. It never reduces an ID to hashValue, so unequal IDs cannot accidentally alias each other.

Root fieldemail
Nested fieldprimaryAddress.country.code
List fieldattendees[attendee-42].name

That path grows when a builder scopes into another subject. Firma has three scope shapes:

ScopePath behaviorLifecycle behavior
Nested modelAppend the parent field.Reuse the child's validation rules inside the root validator.
Optional modelAppend the optional field when present.Retain the last readable value during SwiftUI teardown, ignore stale writes, then remove descendant errors and focus registrations.
Identified collectionAppend the list field and the element's real ID.Return no builder for a stale or foreign ID; reconcile navigation when IDs reorder or disappear.
 
ForEach(event.attendees) { attendee in
  if let row = form.scope(\.attendees, id: attendee.id) {
    row.textField(for: \.name, label: "Attendee name")
    row.stepper(for: \.ticketCount, range: 1...8)
  }
}

The element ID, not its current array index, stays in the path. Move the first attendee to the bottom and its value, error, and focus identity all move with it.

Part 5Dynamic lists without index identity

An editable list is where ordinary form abstractions tend to reveal their seams. Rows can be inserted, deleted, and reordered while validation errors and keyboard focus are still active. If a field is identified by an array offset, moving one element can silently attach another element's error to it.

Firma makes the collection relationship explicit. Each row is its own @FormObject model with a stable Identifiable.ID; the parent stores those models in IdentifiedArrayOf. Parent validation can attach an error to the collection itself, then compose every current row's rules in collection order.

import Firma
import Foundation
import Observation

enum EventError: LocalizedError {
  case attendeeNameRequired
  case needsAttendee

  var errorDescription: String? {
    switch self {
    case .attendeeNameRequired: "Enter an attendee name"
    case .needsAttendee: "Add at least one attendee"
    }
  }
}

@Observable @FormObject
final class AttendeeForm: Identifiable {
  let id: UUID
  var name = ""
  var ticketCount = 1

  init(id: UUID = UUID()) {
    self.id = id
  }

  func validate() {
    if name.isEmpty {
      addError(EventError.attendeeNameRequired, for: \.name)
    }
  }
}

@Observable @FormObject
final class EventForm {
  var title = ""
  var attendees: IdentifiedArrayOf<AttendeeForm> = []

  func validate() {
    if attendees.isEmpty {
      addError(EventError.needsAttendee, for: \.attendees)
    }
    validate(\.attendees)
  }
}

validate(\.attendees) is not implied by rendering the rows. It is an explicit composition choice in the parent model, just like validating a nested address. Firma walks the collection in its current order, evaluates each child's validator at attendees[elementID], and merges the results into the root snapshot.

Render the current row, not a captured copy

The view iterates over the model for SwiftUI identity, then asks the root builder for the row currently stored under that ID. The scoped builder owns the complete path and live read/write access back into the collection.

ForEach(model.attendees) { attendee in
  if let row = form.scope(\.attendees, id: attendee.id) {
    row.textField(for: \.name, label: "Attendee name")
    row.stepper(for: \.ticketCount, label: "Tickets", range: 1...8)

    Button("Remove", role: .destructive) {
      model.attendees.remove(id: attendee.id)
    }
  }
}

Button("Add attendee") {
  model.attendees.append(AttendeeForm())
}

The optional return from scope(_:id:) is important. SwiftUI may briefly evaluate an old row after its element has been removed. Firma checks the current collection before creating the builder, retains the last value only long enough for safe teardown, and ignores writes once the ID is gone. A stale view therefore cannot recreate a phantom element.

MutationWhat stays stableWhat Firma updates
InsertEvery existing row keeps the same element ID and descendant paths.The new ID receives its own path; rendering and the next validation pass place it at its collection position.
RemoveUnrelated rows keep their bindings, errors, and focus identities.Collection observation prunes focus/order state; the next validation pass replaces list descendants so removed-row errors disappear.
ReorderEach row keeps its value and any error under attendees[id].Keyboard order follows the observed ID sequence; the next validation pass produces errors in that same order.

Collection errors and row errors are different queries

The path attendees belongs to the collection field; attendees[id].name belongs to one row. That lets a screen present both levels without flattening them:

// Only an error attached directly to the list, such as "Add an attendee".
let listError = form.error(for: \.attendees)

// The list error plus every error produced by every current row.
let allAttendeeErrors = form.errors(for: \.attendees)
IDs are form identity. Use an ID that is unique and stable for the row's lifetime. An array index describes a temporary position; changing a model's ID after rendering changes its form identity. IdentifiedArrayOf makes uniqueness and ID-based mutation explicit at the storage boundary.

The focus lab later in this post demonstrates the same rule from the keyboard's point of view: reorder an attendee and its field moves through the navigation sequence without losing which row it belongs to.

Part 6A form engine that does not own the layout

FirmaContent installs the validator and focus coordinator, then hands a FirmaBuilder to the content closure. It does not insert a Form, List, ScrollView, or grid. The application keeps complete control over hierarchy and styling.

FirmaContent(editing: $profile) { form in
  ScrollView {
    LazyVStack(alignment: .leading) {
      form.textField(for: \.name, label: "Name")
      form.textField(for: \.email, label: "Email")
      form.toggle(for: \.receivesUpdates, label: "Updates")

      form.submitButton("Save") {
        save(profile)
      }
    }
  }
}

The built-in text field, toggle, and stepper wire a binding, error, stable ID, focus preference, and accessibility metadata together. A custom control can join the same system through control(for:focusable:):

form.control(for: \.referralCode, focusable: true) { field in
  VStack(alignment: .leading) {
    TextField("Referral code", text: field.$value)
      .focused(field.$focus, equals: field.id)

    if let error = field.error {
      Text(error.localizedDescription)
    }
  }
}

Marking a custom control focusable is a contract: the view must attach the supplied focus binding and ID. Setting focusable: false still gives the control its binding and error without putting it into keyboard navigation.

Part 7Fresh validation passes

A validation pass resets stale state, runs the model's rules, merges nested results, and returns an immutable ValidationResult. Its error dictionary supports exact and descendant lookup; its ordered errorPaths preserves production order so submission can choose a deterministic first error.

ValidationResultnot run
A pass starts by clearing the previous snapshot.

Try an invalid referral, validate, then remove it and validate again. The old referral error disappears because every root pass starts from an empty state. Scoped validation is more surgical: it replaces errors below that child path while preserving errors elsewhere in the form.

A snapshot, not live truth. ValidationResult does not change when the model changes. Call validation() or runValidation() for a new result. That makes submission decisions deterministic and prevents a previously returned value from changing under its consumer.

Part 8Focus that survives lazy content

Rendered controls report their frames through a preference key. Firma sorts those entries into visual order, then reconciles that mounted snapshot with a longer-lived focus order. The distinction matters: a lazy stack can unmount a row without removing it from the model or from keyboard navigation.

Identified scopes separately observe collection order. Reordering rows rewrites the corresponding slices of the focus sequence by element ID; removing a row prunes its paths and any descendant list observations.

Interactive — identified rows and keyboard order
The dashed row exists in the model and focus order but is currently outside the lazy container's mounted region.

When navigation targets an unmounted field, the coordinator stores a pending request, asks ScrollViewReader to reveal the candidate, and waits for the next mounted-field preference update before assigning FocusState. If a candidate disappears during that process, it moves on rather than focusing a stale path.

The same mechanism powers invalid submission. Firma takes errorPaths in rule order, filters them through the known focus sequence, and requests the first focusable error. A date picker marked non-focusable can still have an error; it simply will not block the search for the next keyboard destination.

Part 9Submission, styling, and sharp edges

1 · ValidateStart a fresh pass for the current scope.
2 · DecideContinue only when the snapshot is valid.
3 · RecoverScroll to and focus the first usable error.
4 · SubmitRun the sync or MainActor async action.

submitButton and asyncSubmitButton both use that flow. The asynchronous variant owns a task and prevents repeated taps from producing overlapping successful submissions. Validation itself remains synchronous and main-actor bound, which is appropriate for local form rules; server-side checks still belong in the submission action or in application state around it.

FirmaStyle is intentionally narrow. It configures the convenience controls' error and focused-label colors through the environment. It is not a theme engine, and FirmaContent does not impose spacing, sections, button styles, or navigation.

Reference semantics are part of the design. @FormObject supports classes, requires @Observable, and generates reference-writable field access. Value-type form models are outside this API. Dynamic lists use IdentifiedArrayOf so an element's stable ID, not an array offset, owns its descendant paths.

The sample app is unusually valuable here because it tests behavior that unit tests cannot fully establish: custom focus, keyboard Previous/Next controls, lazy scrolling, optional teardown, identified reordering, synchronous submission, and cancellation-safe asynchronous submission.

Part 10API reference & verdict

@FormObject and Firma

Generate typed field metadata, one validator, and protocol conformance for an observable class. Implement validate(); use addError(_:for:) and validate(_:) for local and nested rules.

FirmaContent(editing:builder:)

Install the root validation and focus environment without choosing a visual container. The closure receives FirmaBuilder<F>.

FirmaBuilder values and errors

binding(for:) returns a native binding. error(for:) reads one exact path. errors(for:) returns every error at or below a field. validation() returns a snapshot; validate() returns its Boolean validity.

Scopes and controls

scope(_:) enters nested or optional models; scope(_:id:) enters an identified row. Convenience APIs cover text fields, toggles, and steppers. control(for:focusable:content:) exposes value, error, path, and optional focus plumbing to any SwiftUI view.

Focus and submission

focus(on:) requests a field and may complete after lazy content mounts. Submit buttons validate first, focus the first rendered error on failure, and run sync or async actions only on success.

Firma's strongest choice is refusing to treat validation, rendering, and focus as separate subsystems. A macro-generated field becomes a path; that path becomes a binding, an error address, a scroll target, and a place in the keyboard order. Nested models and identified collections extend the path rather than inventing new coordination.

That makes the package especially useful for forms whose complexity is structural: optional sections, reusable child models, editable lists, custom controls, lazy containers, and error-first submission. Small forms still stay small. Large forms gain one consistent identity model instead of accumulating adapters between five almost-compatible ones.
