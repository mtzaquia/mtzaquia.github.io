Departure: routing for SwiftUI that starts from intent, not from state

A close look at mtzaquia/departure — a routing framework where screens declare the destinations they own, and any part of your app can simply ask for a route.

Part 1The problem with SwiftUI navigation at scale

SwiftUI gives you navigation primitives — NavigationStack, .sheet, .fullScreenCover — but no navigation architecture. Every presentation is a boolean or an optional binding owned by some view, and the code that wants to navigate must know exactly which view owns that binding, and how it presents.

That coupling is fine for three screens. It falls apart when:

  • A deep link or push notification needs to open a screen from outside the view hierarchy entirely.
  • A flow must be guarded — “show settings, unless the user is logged out, in which case show login first, then come back.”
  • A TabView is involved, and each tab needs its own push path while modals stay global.
  • Something genuinely urgent — an outage banner, a forced re-auth — must appear above whatever the user is doing, no matter how deep they are.

Departure’s answer is a single inversion: the screen that owns a presentation declares it; the code that starts a flow only requests it. The framework connects the two at runtime by walking its own model of your navigation state and finding the closest eligible owner.

// Anywhere in the app — a button, a deep link handler, a view model:
await router.present(SettingsRoute())

No bindings, no knowledge of where or how settings appears. That is decided by whichever screen declared it.

Part 2The model: routes, declarations, ownership

Departure has exactly three moving parts. Everything else in the framework is machinery to make these three compose.

ConceptWhat it isYou write
Route A value describing a destination. It builds its own view and can veto or redirect its own presentation. struct ProfileRoute: Route
Declaration A statement by a view: “I own the presentation of this route type, in this style.” .routes { Push(ProfileRoute.self) }
Router An @Observable engine, installed once via WithRouter, read from the environment everywhere. @Environment(Router.self)

A route is just a struct with a destination() view builder:

struct ProfileRoute: Route {
  let userID: String

  func destination() -> some View {
    ProfileView(userID: userID)
  }
}

And a view claims ownership with a result-builder modifier. The same route type may be declared in several scopes — the nearest owner to the active route wins, so a route can be a push in one context and a sheet in another:

struct HomeView: View {
  @Environment(Router.self) private var router

  var body: some View {
    Button("View profile") {
      Task { await router.present(ProfileRoute(userID: "42")) }
    }
    .routes {
      Push(ProfileRoute.self)      // pushes onto the nearest NavigationStack
      Sheet(SettingsRoute.self)    // modal, wrapped in its own NavigationStack
      Cover(OnboardingRoute.self)  // full-screen cover
    }
  }
}

Presentation styles

DeclarationPresentationNotes
PushPushes onto the nearest NavigationStackMust be declared inside a stack.
SheetModal sheetWraps destination in a NavigationStack by default; opt out with providesNavigation: false.
CoverFull-screen coverSlide transition by default; transition: .fade gives a cross-dissolve rendered in a detached host.
Design note. Because a Route’s default identity is its type (ObjectIdentifier(Self.self)), most routes need zero identity boilerplate. Conform a route to Equatable when its value identifies a destination — then, if an equal route already sits on the active path, Departure unwinds to it instead of stacking a duplicate. That single rule prevents the familiar “tapped the notification twice, got two copies of the screen” bug.

Part 3What happens inside present()

The most interesting part of Departure is what happens between await router.present(route) and the destination appearing. It’s a small pipeline, and each stage is a place you can hook in. Click each stage (or use the button) to walk through it.

Click a stage above, or press Step through, to see what the router does at each point.

Stage 1 is where guarded routes live. Any route can implement resolveRoute() and decide, asynchronously, whether it should present at all:

struct ProtectedSettingsRoute: Route {
  let isLoggedIn: Bool

  func resolveRoute() async -> RouteResolution {
    isLoggedIn ? .allow : .reroute(LoginRoute())
  }

  func destination() -> some View { SettingsView() }
}

The three possible resolutions map directly onto the enum in Route.swift:

ResolutionRouter behavior
.allowProceed to owner lookup and present. This is the default implementation.
.reroute(any Route)Present a different route instead. The replacement is also resolved — you own the responsibility of not creating a loop.
.dropSilently ignore the request.
Keep resolution fast. Routing suspends until resolveRoute() returns. It’s the right place for a cheap state check (“am I logged in?”), not for a network round trip.

Two things worth noticing while you play:

  • Duplicate protection. With ProfileRoute already on the path, presenting it again doesn’t stack a copy — the router unwinds back to the existing one. That’s the Equatable rule from Part 2 in action.
  • present() is async and settles. The real API returns after the router’s state is updated, so you can sequence navigation (await present(a); await present(b)) without the timing hacks SwiftUI normally forces on you.

Part 5Unwinding and returning values

Dismissal in Departure is as declarative as presentation. Instead of chaining dismiss() calls up a modal stack, you unwind to a target:

await router.unwind(to: .root)             // everything, all branches
await router.unwind(to: .nearestBranch)    // current tab back to its root
await router.unwind(to: .topmostAncestor)  // one level below the topmost scope
await router.unwind(to: .id("settings-flow"))  // a tagged scope

Tagging a scope makes it a stable target — the tag lives on the same .routes modifier you already use for declarations:

SettingsFlowView()
  .routes(id: "settings-flow") {
    Push(AdvancedSettingsRoute.self)
  }

unwind(to:) returns a Bool — whether a target was found — so multi-step flows can be written as plain, awaitable sequences:

if await router.unwind(to: .id("settings-flow")) {
  await router.present(LoginRoute())
}

Typed payloads on the way out

Dismissal can also return a value. A presenting scope installs a typed UnwindHandler; the presented flow dismisses with a payload; the handler receives it — with the types checked for you:

// The presenter listens:
.hooks {
  UnwindHandler(EditorRoute.self, expecting: SaveResult.self) { result in
    showToast(for: result)
  }
}

// The editor, on its way out:
await unwindRoute(payload: SaveResult.saved)

And for local dismissal from inside a route’s subtree, @Environment(\.unwindRoute) is the scope-bound equivalent of SwiftUI’s dismiss — with the nice property that SwiftUI’s own dismiss() (or an interactive swipe) still flows through the same unwind path and fires a matching handler.

Part 6Actions: intent that routes itself

The most unusual idea in Departure. An Action is a unit of user intent that must run in the right place — and can ask the router to get it there first:

struct SaveDraftAction: Action {
  func attemptAction(in context: ActionContext) async throws(ActionInvocationError) {
    guard context.isRunning(in: EditorRoute.self) else {
      throw .reroute(EditorRoute())
    }
    // Save the draft.
  }
}

await router.perform(SaveDraftAction())

If the action throws .reroute(route), Departure presents that route through the normal pipeline, then retries the action exactly once in its new scope. “Save the draft” works whether or not the editor is currently on screen — the action carries its own precondition.

Scopes can also intercept actions passing through them — wrap them, observe them, or consume them entirely:

.hooks {
  ActionInterceptor(SaveDraftAction.self) { invocation in
    do {
      try await invocation()   // run the original; omit to consume it
    } catch {
      // Show a save error.
    }
  }
}

Only the active route scope participates in interception, so a screen buried under a sheet can’t hijack an action that no longer belongs to it. It’s middleware for navigation-adjacent intent, with the scoping rules already solved.

Part 7Branches: tabs done right

Tab bars are where most SwiftUI routing schemes break down: each tab needs its own independent push path, modals should be shared, and routing to a screen owned by another tab should switch tabs first. Departure models this with branches.

TabView(selection: $tab) {
  NavigationStack { HomeView().routeBranch(AppTab.home) }
    .tag(AppTab.home)

  NavigationStack { WalletView().routeBranch(AppTab.wallet) }
    .tag(AppTab.wallet)
}
.routes(branch: $tab) {
  Cover(LoginRoute.self)                             // owned by the container
  Branch(.home)   { Push(HomeDetailRoute.self) }     // discovery map per tab
  Branch(.wallet) { Sheet(TransactionRoute.self) }
}

The container’s Branch(...) entries are a discovery map: they let the router route into a tab that has never been built (a lazy tab), because the map — not the view — knows what lives there. Try it below: present TransactionRoute while sitting on the Home tab.

Tabs can also be switched directly — each branch keeps its own push path either way.

The rules that make this predictable:

  • Branches keep independent push paths, but share modal presentations — a sheet from one branch replaces a live modal from another.
  • Declarations outside any Branch(...) belong to the container — the natural home for cross-tab flows like login.
  • Explicit .routes { } on a .routeBranch(...) view take precedence over the container’s map, so the same view works inside and outside a branched container.

Part 8Priority: windows above the app

Some flows can’t wait for the navigation stack to be in a convenient state — forced re-authentication, a critical outage screen. Departure lets a declaration claim elevated priority, and backs it with real UIKit windows (PassThroughWindow and a window bridge live in the framework’s UIKit layer):

.routes {
  Sheet(ProfileRoute.self)                          // .normal
  Cover(LoginRoute.self, priority: .high)
  Cover(SystemOutageRoute.self, priority: .critical)
}
PriorityUseMechanics
.normalEveryday navigationLives in the app’s own hierarchy.
.highImportant flows above normal navigationOwn window above the app.
.criticalUrgent flows above everything elseOwn window above even .high.

The rules discourage overuse: elevated flows are for interruption, not stacking. A high/critical request from normal content replaces any active presentation at that priority; requesting from an equal-or-higher context degrades to plain local routing; and a lower-priority request from outside an active elevated context simply doesn’t present.

Because elevated presentations render in a separate window, custom environment values don’t flow in automatically. WithRouter takes a windowDestination closure for exactly this — forward what your destinations need: 
WithRouter {
  AppRoot()
} windowDestination: { destination, environment in
  destination.environment(\.myCustomKey, environment.myCustomKey)
}

Where it shines, where to be careful

👍 Shines⚠️ Watch for
Deep links and notifications: fire a route from anywhere; guards and branch selection do the rest. Requires buying into the model app-wide — mixing Departure with manual .sheet bindings for the same flows will fight it.
Awaitable, sequenceable navigation (await present…; await unwind…) replaces dispatch-after hacks. resolveRoute() suspends routing — keep guards to cheap local checks.
Typed unwind payloads and action interception replace the closures and NotificationCenter plumbing teams usually write for these flows. Reroute chains (guards + action reroutes) are your responsibility to keep loop-free.
Priority windows solve the “show this above everything” problem without hacky window code in the app. Detached hosts (elevated priority, fade covers) need environment forwarding via windowDestination.

At its core, Departure treats navigation as a semantic graph the app can query, not a pile of booleans views happen to hold. Once presentation ownership is declarative, everything downstream — guarding, deduplication, cross-tab routing, typed dismissal, elevated windows — becomes a router policy instead of a per-screen puzzle. If your SwiftUI app has grown past the point where you can hold the navigation state in your head, this is a framework worth a weekend spike.
