Ensemble: keep the data, change the phase

A source-level tour of Ensemble 1.0 — a SwiftUI library that separates the latest useful value, asynchronous source lifetime, and each view's loading and failure policy.

Part 1The latest event is not the latest useful UI

A refresh starts while yesterday's account summary is still perfectly useful. Then the request fails. The newest event is an error, but replacing the summary with a full-screen failure would make the interface less informative than it was before the request began.

Many asynchronous view models make that replacement unavoidable. One enum carries idle, loading, success(Value), and failure(Error). Moving to loading throws away the successful case; moving to failure throws it away again. The view can preserve a second copy, but now cache ownership and request state have become application-specific coordination.

The problem gets sharper on a composed screen. Activity may be waiting for its first response, the account card may be refreshing cached data, and a small tip may have failed. One screen-wide state either blocks sections that are ready or hides distinctions that matter.

The central idea. Ensemble keeps the current lifecycle phase and the latest successful value as separate facts. A source changes those facts; each AsyncContent decides what they mean for its own piece of UI.

Part 2Give state, lifetime, and rendering different owners

Ensemble's public model is small because it draws three ownership boundaries instead of building one all-purpose loader.

ViewData<Value> Owns presentation state Stores the current phase, latest successful value, retry availability, and an opaque animation token.
ViewDataContext Owns asynchronous work Runs one-shot loads, retains stream factories, and owns at most one binding for each destination.
AsyncContent Owns rendering policy Maps one state into live, cached, placeholder, failure, or empty output without imposing a container. 
final class EntriesViewModel {
  let entries = ViewData<[Entry]>()

  private let context = ViewDataContext()
  private let useCase: EntriesUseCase

  init(useCase: EntriesUseCase) {
    self.useCase = useCase
  }

  func start() {
    context.bind(useCase.values, to: entries)
  }
}

ViewData is observable, so a surrounding view model does not need observation solely to expose it. The context stays alive for as long as its subscriptions should stay alive. The view only receives the state it needs to render.

This separation also keeps source mechanics out of SwiftUI. A list does not need to know whether its values came from one HTTP request, a database observation, or a long-lived stream. It needs to know whether the value is live, retained, or synthetic presentation input.

Part 3Keep the data when the phase changes

ViewData has four phases: empty, loading, success, and failure. Its separate latest property holds only the most recent successful value. Loading and failure change the phase without clearing that value.

What just happened?
phase = .failure(error)

The current request or stream update failed.

What is still useful?
latest = previousResponse

The most recent success remains available for cached presentation.

A seeded value starts in success immediately:

let entries = ViewData(cachedEntries)

// A later binding enters loading, but cachedEntries remains in entries.latest.
context.bind(useCase.values, to: entries)

A placeholder is different. AsyncContentLoadingPolicy.placeholder supplies a value only to the renderer when no success exists; it never installs that value in latest. The distinction prevents skeleton rows from becoming application data.

reset() is deliberately narrow in the other direction. It clears the phase and latest value but does not cancel a load or binding. A later accepted update can fill the state again. Cancellation belongs to the context or to the task awaiting a one-shot load.

Retention is not persistence. Ensemble keeps one successful value in memory. It does not decide how long data is fresh, write a disk cache, merge records, or resolve conflicts. Those remain source and application concerns.

Part 4Make presentation a view-local policy

AsyncContent turns the two stored facts into a view. Without a failure builder, its defaults are intentionally quiet: before the first success, loading and failure render nothing; after a success, both retain the latest value. Supplying a failure builder makes replacement the failure default. A call site can choose cached failure instead, and can opt into placeholders when that region needs them.

AsyncContent(
  viewModel.entries,
  loading: .placeholder(Entry.placeholders),
  failure: .cached
) { entries, source in
  EntryRows(entries: entries)
    .redacted(reason: source == .placeholder ? .placeholder : [])
} failure: { error, retry in
  EntriesFailure(error: error, retry: retry)
}

The success builder receives an AsyncContentSource: .live, .cached, or .placeholder. That gives the view enough context to add a stale badge or redaction without changing the stored value.

Use the lab to answer one question: what does this view render as the lifecycle changes? Start from the seeded value, begin a refresh, fail it, and compare cached failure with replacement failure. Then reset the cache and repeat.

Notice that policies do not mutate the left-hand state. Switching from .cached to .replace changes the output while phase and latest stay put. Two views can therefore render the same ViewData differently without asking the source to publish a different model.

The failure overloads encode one useful constraint in the type system. A view with a failure builder can choose .cached or .replace. A view without one gets a different policy type with .cached or .hidden; it cannot select a replacement view that does not exist.

Part 5Treat failure as an update, not a terminus

A bound Ensemble source is a non-throwing AsyncStream whose elements are Result values. Failure lives inside the sequence rather than ending it.

func values() -> AsyncStream<Result<[Entry], EntryFailure>> {
  AsyncStream { continuation in
    continuation.yield(.failure(.offline))
    continuation.yield(.success(recoveredEntries))
    continuation.finish()
  }
}

The first element moves the destination to failure while preserving any previous success. The second replaces latest and moves back to success through the same subscription. The test suite exercises that exact failure-then-success sequence.

This shape fits sources where outages are observations rather than terminal conditions: reachability-sensitive data, database listeners, or a producer that can recover after authentication changes. A cold request may still finish after emitting one result; Ensemble supports both lifetimes.

The stream factory must return promptly. bind calls it synchronously on the main actor. Networking, observation, and other expensive work belong in the stream's producer, not before the factory returns.

A one-shot throwing operation uses load(_:to:) instead. Its error becomes presentation state rather than being rethrown, and cancellation does not become a visible failure. A load does not install a retry action because Ensemble has no factory to repeat; retry belongs to a bound source.

Part 6Reload according to the source's lifetime

“Reload” means different things to different sources. A cold source should create a new request. A hot source should usually keep listening and ask its existing producer to publish again. Ensemble makes that choice part of the binding.

Cold source · .resubscribe cancel old stream → call factory → consume new stream

The default for request-per-subscription sources. Late values from the replaced registration are ignored.

↔
Hot source · .refresh(action) keep stream → mark loading → signal producer

Preserves an active subscription. If it already completed, Ensemble reattaches before calling the action.

// One request for every subscription.
context.bind(useCase.values, to: entries)

// One long-lived stream; tell its producer to refresh.
context.bind(
  useCase.values,
  to: entries,
  reload: .refresh(useCase.reload)
)

context.reload(entries) and the optional retry action passed to a failure builder use the same configured behavior. .disabled installs neither. The UI does not need a second switch statement to recreate source-specific logic.

A completed hot sequence is a subtle hard case. On refresh, the context creates the next sequence before invoking the refresh action, ensuring the producer has a consumer for the result. The source still owns its buffering contract: if it deliberately drops values before iteration begins, it must coordinate that publication itself.

Part 7Make cancellation and competing work predictable

A load and a binding may feed the same destination at the same time. Ensemble does not assign permanent priority to either one; accepted updates apply in arrival order. The latest success replaces latest, while a later loading or failure update preserves it.

Bindphase becomes loading
Loada one-shot operation also starts
Stream: 1success installs value 1
Load: 2later success installs value 2
Stream: 3binding remains active and installs value 3

The implementation protects this rule with identity and loading tokens. Each destination has at most one registered binding. Rebinding cancels the old task, and its loop checks that its registration is still current before applying another value. A cancelled load only settles the loading transition it started; it cannot overwrite a newer bound success.

Ownership completes the model:

The context retains the stream factory and any refresh closure. A bound method retains its instance. That is useful when a dedicated use case should share the context's lifetime, but passing a method on an owner that also retains the context creates a cycle. Use a separate source object or an intentional weak capture in that design.

Part 8Compose independent regions, not one screen-wide switch

AsyncContent does not insert a List, Section, stack, or navigation container. Its output can be a whole screen, but it can just as naturally be one section inside a stable parent.

List {
  AsyncContent(
    viewModel.activity,
    loading: .placeholder(Activity.placeholders),
    failure: .replace
  ) { activity, source in
    ActivitySection(activity: activity, source: source)
  } failure: { error, retry in
    ActivityFailure(error: error, retry: retry)
  }

  AsyncContent(
    viewModel.account,
    loading: .cached,
    failure: .cached
  ) { account, source in
    AccountSection(account: account, source: source)
  } failure: { error, retry in
    AccountFailure(error: error, retry: retry)
  }
}

Now activity can show placeholders while the account section retains its last value through a failed refresh. Each region owns a policy proportional to its importance; neither waits for the other.

There is one SwiftUI lifecycle trap worth naming. Empty state renders EmptyView, so an initial .task attached directly to that output is not a reliable place to begin the binding. Attach startup work to a stable ancestor such as the surrounding List or ZStack.

For animation, ViewData.animationValue advances on every accepted presentation change, including phase transitions, values, errors, reset, and changes to retry availability. It does not compare or retain Value, so non-Equatable models can still drive SwiftUI's value-scoped animation.

AsyncContent(viewModel.entries) { entries, _ in
  EntryRows(entries: entries)
}
.animation(.default, value: viewModel.entries.animationValue)

Lifecycle diagnostics are opt-in with Ensemble.debug = .normal or .trace. Debug-build logs include a destination identity so independent regions can be separated, but they do not log loaded values.

Part 9Sharp edges, API reference, and verdict

Ensemble makes asynchronous presentation smaller by refusing to collapse three decisions into one enum. That does leave real application work outside the package.

Ensemble is a strong fit when…The application still decides…
Useful data should survive refresh and failure.Freshness rules, persistence, invalidation, and domain-specific empty states.
Several SwiftUI regions load and recover independently.How those regions coordinate when the product requires an all-or-nothing result.
A source may recover after emitting a failure.Buffering, backpressure, and upstream scheduling inside the producer.
Cold and hot sources need different reload behavior.Whether a source truly represents a new subscription or an existing producer.
Main-actor presentation is part of the contract. The package uses Swift 6.3, enables main-actor default isolation, and supports iOS 17 and macOS 14. Source producers remain responsible for their own scheduling, but state coordination and stream factory calls cross the main actor.

ViewData<Value>

Observable, main-actor presentation state with phase, retained latest, and animationValue. Initialize empty or with a successful value. reset() clears presentation without cancelling work.

ViewDataContext

load(_:to:) applies one throwing async operation. bind(_:to:reload:) consumes a non-throwing result stream and retains its factory. reload, cancel, and cancelAll act on owned bindings.

AsyncContent and its policies

Loading can be hidden, cached, or use a placeholder. With a failure builder, failure can retain cached content or replace it; without one, failure can retain cached content or stay hidden. The content builder receives a live, cached, or placeholder source marker.

ViewDataReloadBehavior and retry

.resubscribe replaces a cold source, .refresh(action) signals a hot producer, and .disabled exposes no reload. A reload-enabled binding supplies the same behavior through an optional ViewDataRetryAction to failure UI.

Adopt Ensemble when the hard part of asynchronous SwiftUI is no longer starting a task, but deciding what should remain useful while that task changes phase. Its best design choice is modest and consequential: a loading event can be current without being allowed to erase the last good answer.

That makes retention explicit, recovery routine, and independently loading UI genuinely independent. In return, the application must accept the ownership model: keep contexts alive, classify sources honestly, attach startup work to stable views, and make persistence a separate concern.
